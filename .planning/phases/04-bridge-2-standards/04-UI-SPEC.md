# Phase 4 — UI Design Contract

> Visual and interaction contract for Bridge 2 (Standards). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Project Context

This is a **CLI-only Node.js library** (`milieu-cli`). There is no browser UI, no React, no CSS, no design system. The "UI" is:
1. **Typed data structures** (`Check`, `BridgeResult`) produced by this phase -- consumed by terminal rendering (Phase 6) and JSON output (Phase 7)
2. **User-facing strings** in `Check.label`, `Check.detail`, and `Check.data` fields

This UI-SPEC defines the **copywriting contract** for Bridge 2 check functions. All strings declared here are the authoritative source for implementation. The contract follows the established Bridge 1 patterns exactly (see source: `src/bridges/reachability/`).

### Check ID and Label Convention

Following Bridge 1's established pattern:
- **id:** lowercase_snake_case, descriptive noun or noun phrase (e.g., `https_available`, `robots_txt`)
- **label:** Title Case, human-readable short label (e.g., `HTTPS Available`, `robots.txt`)
- **detail (pass):** Sentence fragment describing what was found (e.g., `robots.txt found with 12 rules`)
- **detail (partial):** Sentence fragment describing the partial state (e.g., `Partially restricted`)
- **detail (fail):** Sentence fragment describing the failure (e.g., `No robots.txt found`)

### Check Definitions (8 checks)

#### Check 1: OpenAPI Spec (STND-01, STND-02)

| Field | Value |
|-------|-------|
| id | `openapi_spec` |
| label | `OpenAPI Spec` |
| detail (pass, JSON) | `OpenAPI {version} found with {N} endpoints` |
| detail (pass, YAML) | `OpenAPI {version} found (YAML format)` |
| detail (partial) | `OpenAPI spec detected but not parseable` |
| detail (fail) | `No OpenAPI spec found` |
| data (pass) | `{ version: string, specType: "openapi" \| "swagger", endpointCount: number, path: string }` |

Status mapping:
- **pass:** Valid JSON spec found at any of 9 paths, contains `openapi` or `swagger` key, Content-Type is not text/html
- **partial:** YAML spec detected (Content-Type is yaml, body matches `/^(openapi|swagger):/m`) but cannot fully parse without YAML dependency
- **fail:** No valid spec at any of 9 probed paths

#### Check 2: llms.txt (STND-03)

| Field | Value |
|-------|-------|
| id | `llms_txt` |
| label | `llms.txt` |
| detail (pass) | `llms.txt found ({N} bytes)` |
| detail (partial) | `llms.txt found but missing H1 header` |
| detail (fail) | `No llms.txt found` |
| data (pass) | `{ sizeBytes: number, firstLine: string }` |
| data (partial) | `{ sizeBytes: number, firstLine: string }` |

Status mapping:
- **pass:** HTTP 200, non-empty body, first line matches `/^#\s+\S/`
- **partial:** HTTP 200, non-empty body, first line does not start with H1
- **fail:** HTTP 404, empty body, or HTTP error

#### Check 3: llms-full.txt (STND-04)

| Field | Value |
|-------|-------|
| id | `llms_full_txt` |
| label | `llms-full.txt` |
| detail (pass) | `llms-full.txt found ({N} bytes)` |
| detail (fail) | `No llms-full.txt found` |
| data (pass) | `{ sizeBytes: number }` |

Status mapping:
- **pass:** HTTP 200, non-empty body
- **fail:** HTTP 404, empty body, or HTTP error

Note: No partial state. No H1 requirement (llms-full.txt is a comprehensive dump, not required to follow llms.txt format).

#### Check 4: MCP Endpoint (STND-05)

| Field | Value |
|-------|-------|
| id | `mcp_endpoint` |
| label | `MCP Endpoint` |
| detail (pass) | `MCP endpoint found: {detail}` where detail is from validation |
| detail (partial) | `MCP endpoint found but structure unclear` |
| detail (fail) | `No MCP endpoint found` |
| data (pass) | `{ detail: string }` |

Status mapping:
- **pass:** HTTP 200 at `/.well-known/mcp.json`, valid JSON, contains MCP fields (`serverInfo`+`transport`, OR `mcp_version`/`mcpServers`, OR `tools`/`resources`/`prompts`)
- **partial:** HTTP 200, valid JSON, but no recognized MCP fields
- **fail:** HTTP 404, non-JSON body, or HTTP error

#### Check 5: JSON-LD (STND-06)

| Field | Value |
|-------|-------|
| id | `json_ld` |
| label | `JSON-LD Structured Data` |
| detail (pass, singular) | `1 JSON-LD block found: {type}` |
| detail (pass, plural) | `{N} JSON-LD blocks found: {types joined with ", "}` |
| detail (fail) | `No JSON-LD structured data found` |
| data (pass) | `{ blocks: Array<{ type: string \| string[], context: unknown }> }` |

Status mapping:
- **pass:** 1 or more `<script type="application/ld+json">` blocks with valid JSON containing `@type`
- **fail:** No JSON-LD blocks found in page HTML

Note: No partial state. JSON-LD is either present or absent.

#### Check 6: Schema.org Markup (STND-07)

| Field | Value |
|-------|-------|
| id | `schema_org` |
| label | `Schema.org Markup` |
| detail (pass, JSON-LD) | `Schema.org types found via JSON-LD: {types}` |
| detail (pass, Microdata) | `Schema.org types found via Microdata: {types}` |
| detail (pass, both) | `Schema.org types found via JSON-LD and Microdata: {types}` |
| detail (fail) | `No Schema.org markup found` |
| data (pass) | `{ types: string[], sources: ("json-ld" \| "microdata")[] }` |

Status mapping:
- **pass:** JSON-LD `@context` contains "schema.org" OR Microdata `itemtype` attribute contains "schema.org"
- **fail:** No Schema.org vocabulary detected (other JSON-LD vocabularies do not count for this check)

Note: No partial state. A site with non-Schema.org JSON-LD passes Check 5 (JSON-LD) but fails Check 6 (Schema.org).

#### Check 7: security.txt (STND-08)

| Field | Value |
|-------|-------|
| id | `security_txt` |
| label | `security.txt` |
| detail (pass) | `security.txt found with Contact field` |
| detail (partial) | `security.txt found but missing Contact field` |
| detail (fail) | `No security.txt found` |

Status mapping:
- **pass:** HTTP 200 at `/.well-known/security.txt`, body contains `/^Contact:/mi`
- **partial:** HTTP 200, non-empty body, but no Contact field (RFC 9116 requires Contact)
- **fail:** HTTP 404, empty body, or HTTP error

#### Check 8: ai-plugin.json (STND-08)

| Field | Value |
|-------|-------|
| id | `ai_plugin` |
| label | `ai-plugin.json` |
| detail (pass) | `ai-plugin.json found: {name_for_human}` |
| detail (partial) | `ai-plugin.json found but missing required fields` |
| detail (fail) | `No ai-plugin.json found` |
| data (pass) | `{ nameForHuman: string, schemaVersion: string }` |

Status mapping:
- **pass:** HTTP 200 at `/.well-known/ai-plugin.json`, valid JSON with `schema_version`, `name_for_human`, and `api` fields
- **partial:** HTTP 200, valid JSON, but missing one or more required fields
- **fail:** HTTP 404, non-JSON body, or HTTP error

### Scoring Formula (STND-09)

Same as Bridge 1:
- Pass = 1 point
- Partial = 0.5 points
- Fail = 0 points
- Score = `Math.round((points / maxPoints) * 100)`
- scoreLabel: `>= 80` = "pass", `>= 40` = "partial", `< 40` = "fail"

8 checks total. Maximum score = 100 (all pass). A typical website with security.txt and JSON-LD but no OpenAPI, llms.txt, MCP, or ai-plugin scores approximately 25 (2/8 = 25%, scoreLabel = "fail").

### Error State Copy

When an HTTP probe fails with a non-404 error (timeout, DNS, connection refused):
- Use the error message from the `HttpError` object directly in `detail`
- Status: `"fail"` (not `"error"` -- reserve "error" for unexpected exceptions)
- Follow Bridge 1 pattern: `detail: result.error.message`

When Bridge 1 did not store pageBody in ctx.shared (page was unreachable):
- JSON-LD check: `detail: "No JSON-LD structured data found"`, status: `"fail"`
- Schema.org check: `detail: "No Schema.org markup found"`, status: `"fail"`
- Rationale: These checks depend on page HTML. If the page is unavailable, there is no data to extract. This is not an error condition -- it is simply "not found."

### Data Field Conventions

Following Bridge 1 patterns observed in the codebase:
- `data` is `Record<string, unknown> | undefined`
- Only set `data` when there is structured information to surface in JSON output
- Use camelCase for data field keys (e.g., `ruleCount`, `sizeBytes`, `endpointCount`)
- Do not set `data` on fail results unless there is meaningful diagnostic information

## Shared Context Contract

Bridge 2 depends on Bridge 1 setting `ctx.shared.pageBody` (page HTML string). This requires a modification to Bridge 1's `runReachabilityBridge()` function:

```
// After page GET response in Bridge 1:
if (pageResponse.ok) {
  ctx.shared.pageBody = pageResponse.body;
  ctx.shared.pageHeaders = pageResponse.headers;
}
```

Bridge 2 sets `ctx.shared.openApiDetected` (boolean) for Bridge 3 reuse (SEP-01).
